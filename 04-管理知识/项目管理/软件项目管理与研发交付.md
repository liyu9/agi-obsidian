# 软件项目管理与研发交付

## 10.2 软件研发交付的几种模式

### 10.2.1 敏捷开发模式

#### 敏捷核心价值观
- 个体和互动高于流程和工具
- 可工作的软件高于详尽的文档
- 客户合作高于合同谈判
- 响应变化高于遵循计划

#### Scrum核心角色

| 角色 | 说明 |
|------|------|
| Product Owner | 产品负责人，负责产品需求优先级排序 |
| Scrum Master | 敏捷教练，协调团队工作 |
| 开发团队 | 负责交付 |

#### Scrum核心活动

| 活动 | 频率 | 说明 |
|------|------|------|
| Sprint Planning | 每迭代开始 | 计划本迭代工作内容 |
| Daily Scrum | 每天 | 站会同步进度 |
| Sprint Review | 迭代结束 | 展示成果 |
| Sprint Retrospective | 迭代结束 | 复盘总结 |

#### Sprint迭代周期
- 典型周期：2-4周
- 每个Sprint产出可交付的产品增量

### 10.2.2 瀑布模式

#### 特点
- 阶段分明：需求→设计→开发→测试→上线
- 自上而下按固定次序执行
- 适合大型复杂项目

#### 适用场景
- 传统企业软件
- 需求稳定的项目
- 规范化管理要求

### 10.2.3 双周迭代模式

#### 典型双周迭代流程

| 阶段 | 时间 | 说明 |
|------|------|------|
| 需求评审 | W1D1-W1D2 | 产品经理评审需求，研发参与 |
| 迭代计划 | W1D3 | 确定本迭代工作内容 |
| 技术方案设计 | W1D5-W2D1 | 研发人员进行技术方案设计 |
| 开发实施与测试 | W2D2-W4D4 | 开发、测试阶段 |
| 上线 | W4D5 | 集中上线 |

#### 关键实践
- **集中上线**：将一系列功能打包一次性上线
- **故事看板**：可视化任务跟踪
- **每日站会**：同步信息，快速决策

### 10.2.4 双周迭代模式的局限性

| 局限 | 说明 |
|------|------|
| 最小功能集合问题 | 复杂功能无法在一个迭代内完成 |
| 跨端项目协调难 | 多团队协作难以协同 |
| 工时预估不准 | 复杂系统难以快速准确预估 |

### 10.2.5 瀑布vs敏捷

| 维度 | 瀑布模式 | 敏捷模式 |
|------|----------|----------|
| 适用场景 | 需求稳定的大型项目 | 需求变化快的小中型项目 |
| 交付方式 | 一次性完整交付 | 分批次增量交付 |
| 响应变化 | 困难 | 容易 |
| 客户反馈 | 最后 | 持续 |

#### 敏捷的典型案例：交通工具演进
1. 滑板 → 快速满足基本需求
2. 滑板车 → 改进
3. 自行车 → 进一步改进
4. 摩托车 → 接近目标
5. 小汽车 → 最终方案

#### 辩证观点
- 敏捷不是万能的
- 业务型软件产品不适合纯敏捷
- 实践中可以融合使用
- 找到适合自己的模式最重要

## 10.3 项目过程的分析与优化

### 10.3.1 基础数据准备

#### 需求从提出到上线的关键过程

| 阶段 | 说明 |
|------|------|
| 需求提出 | 记录需求来源 |
| PRD编写 | 开始时间、结束时间 |
| 研发 | 开发开始、提测、结束时间 |
| 测试 | 测试开始、结束时间 |
| 上线 | 实际上线时间 |

#### 数据准确性保证
- 持续培养团队习惯
- 将数据分析纳入团队检查
- 通过自动化系统记录时间节点

### 10.3.2 工作量分析

#### 关键指标

| 指标 | 说明 |
|------|------|
| 需求数量 | 按时交付的需求数量 |
| 工时投入 | 研发、测试、设计工时 |
| 人均产出 | 人均需求交付量 |
| 工时利用率 | 实际工时/可用工时 |

### 10.3.3 工程效率分析

#### 关键指标

| 指标 | 说明 |
|------|------|
| 需求交付周期 | 从提出到上线的时间 |
| 研发周期 | 开发阶段耗时 |
| 测试周期 | 测试阶段耗时 |
| 按时上线率 | 按计划上线的比例 |

### 10.3.4 工程质量分析

#### 关键指标

| 指标 | 说明 |
|------|------|
| 需求变更率 | 变更的需求比例 |
| 暂停原因分析 | 需求暂停的原因分布 |
| 上线后缺陷率 | 上线后发现的bug数量 |
