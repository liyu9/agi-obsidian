# 设计工程化理念

## 核心定义

设计工程化（Design Engineering）是指将设计稿从传统的"下游效果图"角色，提升为"产品工程（代码）的一部分"的工作理念和实践方法。

## 传统模式 vs 工程化模式

| 维度 | 传统模式 | 工程化模式 |
|------|----------|------------|
| 设计稿定位 | 下游参考效果图 | 产品代码的一部分 |
| 协作流程 | 设计→开发自行实现 | 设计→代码自动生成/复用 |
| 一致性 | 依赖人工对照 | 代码级别保证 |
| 交付速度 | 慢，需反复还原 | 快，组件直接复用 |

## 核心价值

### 1. 高效工作流程
- 设计稿可直接转化为可复用组件
- 减少设计-开发之间的信息损耗
- 缩短迭代周期

### 2. 产品一致性
- UI 组件库统一管理
- 多端一致（Web/iOS/Android）
- 避免手动还原设计导致的差异

### 3. 可靠交付
- 设计规范通过代码强制执行
- 减少"设计稿 vs 实现"的心理落差
- 降低沟通成本

### 4. 适应快速交付
- 快速原型验证
- 组件库支持快速拼装
- 响应需求变化更灵活

## 工程化实践要点

1. **设计系统化**：建立 Design Token / 组件库
2. **工具链打通**：设计工具 → 代码（如 Figma → 代码）
3. **组件化管理**：原子设计思维，组件可组合
4. **文档同步**：设计变更同步代码文档

## 典型工具/技术

| 类型 | 示例 |
|------|------|
| 设计工具 | Figma、Sketch |
| 设计转代码 | Figma DevMode、Copilot |
| 组件库 | React Spectrum、Material Design |
| Design Token | Style Dictionary |

## 产品经理行动建议

- 推动设计-前端共建组件库
- 在 PRD 中明确设计规范要求
- 关注设计系统与代码库的一致性维护
- 推动设计工具与研发工具链打通
