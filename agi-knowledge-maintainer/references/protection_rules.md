# 知识框架防护规则

## 核心防护原则

### 1. 图片资源防护

**规则**：图片资源必须存放在对应模块的 `assets/` 子目录

**执行标准**：
- ✅ 正确：`01-产品通用知识/assets/用户访谈/流程图.png`
- ❌ 错误：`01-产品通用知识/流程图.png`（直接放在模块根目录）
- ❌ 错误：`assets/流程图.png`（放在全局assets目录）

**目录结构**：
```
01-产品通用知识/
├── assets/
│   ├── 用户访谈/
│   │   ├── 流程图_01.png
│   │   └── 示例图_02.png
│   └── 产品设计/
│       └── 原型示例.png
├── 01-用户研究与需求调研/
│   └── 用户访谈方法.md
└── 02-产品设计/
    └── 原型设计指南.md
```

**图片命名规范**：
- 格式：`[主题]_[类型]_[序号].[扩展名]`
- 示例：`用户访谈_流程图_01.png`
- 避免：`image1.png`、`截图.png`（无意义命名）

### 2. 文本保真度防护

**规则**：不得删减、过度概括原文内容

**执行标准**：
- ✅ 正确：保留核心概念完整定义
- ✅ 正确：保留方法论完整步骤
- ✅ 正确：保留重要案例关键信息
- ❌ 错误：删减原文关键段落
- ❌ 错误：过度简化导致信息丢失
- ❌ 错误：改变原文核心观点

**允许的优化**：
- ✅ 格式调整（转换为Markdown）
- ✅ 结构重组（按知识框架分类）
- ✅ 标题优化（更清晰的层级）
- ✅ 链接添加（关联相关知识）

**禁止的操作**：
- ❌ 删除原文段落
- ❌ 改变原文观点
- ❌ 过度概括内容
- ❌ 丢失关键信息

### 3. 目录变更控制

**规则**：任何目录增删必须先输出计划

**执行流程**：
1. **识别需求**：发现需要新增或删除目录
2. **输出计划**：
   ```markdown
   ## 目录变更计划
   
   ### 变更类型
   - [新增/删除/移动/重命名] 目录
   
   ### 变更详情
   - **目标路径**：`XX/YY/新目录/`
   - **变更原因**：[具体原因]
   - **影响范围**：[受影响的文件和目录]
   
   ### 操作步骤
   1. [步骤1]
   2. [步骤2]
   3. [步骤3]
   ```
3. **用户确认**：等待用户确认
4. **执行变更**：确认后执行

**变更类型**：

#### 新增目录
```markdown
## 目录新增计划

### 新增目录
- **路径**：`05-AI通用知识/11-新主题/`
- **原因**：发现新的AI技术主题需要归类
- **影响**：无现有文件影响

### 操作
1. 创建目录 `05-AI通用知识/11-新主题/`
2. 创建子目录 `assets/`（如需要）
```

#### 删除目录
```markdown
## 目录删除计划

### 删除目录
- **路径**：`01-产品通用知识/旧分类/`
- **原因**：该分类已合并到其他目录
- **影响**：需移动3个文件到新位置

### 操作
1. 移动文件到新目录
2. 更新文件内部链接
3. 删除空目录
```

#### 重命名目录
```markdown
## 目录重命名计划

### 重命名目录
- **原路径**：`05-AI通用知识/01-Prompt工程/`
- **新路径**：`05-AI通用知识/01-Prompt设计与优化/`
- **原因**：名称更准确反映内容
- **影响**：需更新相关文件链接

### 操作
1. 重命名目录
2. 更新所有引用链接
3. 更新索引文件
```

### 4. 文件变更控制

**规则**：必须记录文件变更原因

**变更记录格式**：
```markdown
## 文件变更记录

### 变更文件
- **文件路径**：`XX/YY/文件名.md`
- **变更时间**：YYYY-MM-DD HH:MM
- **变更类型**：[新增/修改/删除/移动]

### 变更详情
- **变更原因**：[具体原因]
- **变更内容**：[简要描述变更内容]
- **影响范围**：[受影响的其他文件或链接]

### 变更前后对比
- **变更前**：[简要描述]
- **变更后**：[简要描述]
```

**变更日志位置**：
- 文件：`.trae/documents/知识框架更新记录.md`
- 格式：按时间倒序排列
- 内容：每次变更的完整记录

## 防护执行流程

### Step 1: 变更识别
- 识别需要进行的变更类型
- 评估变更影响范围
- 确定变更必要性

### Step 2: 计划输出
- 输出详细的变更计划
- 说明变更原因和影响
- 提供操作步骤

### Step 3: 用户确认
- 等待用户确认
- 解答用户疑问
- 根据反馈调整计划

### Step 4: 执行变更
- 按计划执行变更
- 处理相关文件和链接
- 更新索引和导航

### Step 5: 记录变更
- 在变更日志中记录
- 更新相关文档
- 通知用户完成

## 特殊情况处理

### 批量文件变更
1. 输出批量变更计划
2. 说明每个文件变更原因
3. 用户确认后批量执行
4. 记录批量变更日志

### 紧急变更
1. 说明紧急原因
2. 快速输出简化计划
3. 执行变更
4. 后续补充详细记录

### 变更回滚
1. 记录回滚原因
2. 恢复原始状态
3. 更新变更日志
4. 通知用户回滚完成

## 防护违规处理

### 发现违规
1. 立即停止操作
2. 输出违规说明
3. 提供修复方案
4. 用户确认后修复

### 违规记录
- 在变更日志中标注违规
- 说明违规类型和原因
- 记录修复措施
- 预防类似违规

## 防护原则总结

### 核心原则
1. **图片规范存放**：assets子目录，命名规范
2. **文本内容保真**：不删减、不改观点、不过度概括
3. **目录变更控制**：先计划、后确认、再执行
4. **文件变更记录**：记录原因、说明影响、更新日志

### 执行原则
1. **先计划后执行**：任何变更先输出计划
2. **用户确认优先**：等待用户确认再执行
3. **完整记录变更**：每次变更都要记录
4. **及时更新关联**：变更后更新相关链接

### 质量原则
1. **准确性**：变更操作准确无误
2. **完整性**：变更记录完整详细
3. **及时性**：变更记录及时更新
4. **可追溯**：所有变更可追溯历史
