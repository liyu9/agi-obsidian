# Prompt 工程最佳实践

## 🎯 核心原则

### 1. KISS 原则（Keep It Simple and Specific）

**要点**：
- 指令简洁明确
- 避免歧义和模糊
- 一个 Prompt 专注一个任务

**示例对比**：
```markdown
❌ 复杂模糊：
"帮我分析一下我们的产品，看看有什么问题，然后给出建议，最好能详细一点，包括用户反馈、功能设计、技术实现等方面"

✅ 简洁具体：
"分析电商 APP 购物车功能的 3 个主要问题：
1. 用户体验问题（基于常见痛点）
2. 功能设计缺陷
3. 技术实现风险

每个问题给出 1 个改进建议。"
```

### 2. 上下文充分原则

**要点**：
- 提供足够背景信息
- 明确目标和约束
- 说明使用场景

**检查清单**：
- [ ] 角色定位清晰
- [ ] 任务目标明确
- [ ] 输入信息完整
- [ ] 输出格式指定
- [ ] 约束条件说明

### 3. 迭代优化原则

**Prompt 迭代流程**：
```
V1 基础版 → 测试 → 发现问题 → V2 改进版 → 测试 → V3 优化版
```

**迭代记录示例**：
```markdown
V1: "生成用户画像"
问题：太笼统，输出不实用

V2: "为 CRM 系统生成销售代表的用户画像"
改进：增加了角色和行业

V3: "为 B 端 CRM 系统生成 3 个销售代表的用户画像，包含：
- 基本信息（年龄、工龄、学历）
- 工作目标（KPI、日常任务）
- 痛点挑战（3-5 个）
- 产品期望（功能、体验）
- 使用场景（时间、地点、频率）"
改进：结构化输出，可落地使用
```

## 📋 Prompt 设计清单

### 设计前准备

- [ ] 明确任务类型（创作/分析/总结/转换）
- [ ] 确定目标受众（专业人士/普通用户/技术人员）
- [ ] 准备必要背景信息
- [ ] 收集参考示例（如有）

### 设计中检查

- [ ] 角色定义是否清晰
- [ ] 指令是否无歧义
- [ ] 约束条件是否合理
- [ ] 输出格式是否明确
- [ ] 长度限制是否必要

### 设计后验证

- [ ] 执行测试（3-5 次）
- [ ] 检查输出一致性
- [ ] 评估输出质量
- [ ] 记录 Token 消耗
- [ ] 收集用户反馈

## 🔧 实用技巧

### 1. 分而治之

复杂任务拆解为多个简单 Prompt。

**示例**：
```markdown
任务：生成完整的产品方案

❌ 单个 Prompt：
"生成一个完整的社交产品方案"

✅ 拆分多个 Prompt：
1. "分析目标用户群体和核心需求"
2. "基于用户需求，设计 5 个核心功能"
3. "为每个功能设计用户旅程图"
4. "评估技术可行性和开发成本"
5. "制定 MVP 版本规划"
```

### 2. 提供样板

给出期望的输出格式示例。

**示例**：
```markdown
请按照以下格式生成用户故事：

格式样板：
"作为 [角色]
我想要 [功能]
以便于 [价值]

验收标准：
- [标准 1]
- [标准 2]"

现在请为"用户登录"功能生成用户故事。
```

### 3. 设定评估标准

让模型自我检查输出质量。

**示例**：
```markdown
完成任务后，请：
1. 检查是否回答了所有问题
2. 评估逻辑是否自洽
3. 指出可能的局限性
4. 给出改进建议

然后输出最终答案。
```

### 4. 利用系统提示

某些平台支持设置系统级提示。

**示例**：
```markdown
System: 你是一位资深产品经理，擅长：
- 用户需求洞察
- 产品架构设计
- 数据分析驱动决策
- 跨团队协作

回答风格：
- 结构化表达
- 数据支撑观点
- 给出可执行建议
- 避免空泛理论

User: [具体问题]
```

## 📊 性能优化

### 减少 Token 消耗

**技巧**：
1. **精简上下文**：只保留必要信息
2. **明确长度限制**："用 200 字概括"
3. **避免重复**：不要反复说明同一件事
4. **使用缩写**：首次定义后使用简称

**对比示例**：
```markdown
❌ 冗余：
"我想让你帮我写一份产品需求文档，这份文档是关于我们要开发的一个新的电商系统的，
这个电商系统主要是面向中小企业的，让他们可以在上面开店卖东西..."

✅ 精简：
"生成 B2B 电商系统 PRD，目标用户：中小企业商家。"
```

### 提升响应速度

**方法**：
1. **降低温度参数**：temperature=0.3（更确定）
2. **限制输出长度**：max_tokens=500
3. **简化任务**：避免过于复杂的要求
4. **使用流式输出**：实时查看进度

## 🚫 常见错误与避免

### 错误 1：指令模糊

```markdown
❌ "分析一下这个功能"
✅ "从用户体验、技术实现、商业价值三个维度分析购物车功能"
```

### 错误 2：缺少约束

```markdown
❌ "介绍一下 AI 技术"
✅ "用 500 字向非技术人员介绍机器学习，避免数学公式，多用生活类比"
```

### 错误 3：过度复杂

```markdown
❌ "生成产品方案，包括市场分析、用户调研、竞品分析、功能设计、
    技术架构、运营策略、财务预测、风险评估..."
✅ "先生成产品方案的核心功能部分，其他内容后续补充"
```

### 错误 4：忽略示例

```markdown
❌ 只描述要求，不给示例
✅ "参考以下格式：[示例]，现在生成类似内容"
```

### 错误 5：不验证输出

```markdown
❌ 直接使用第一次输出
✅ 测试 3-5 次，检查一致性和质量
```

## 🔄 Prompt 版本管理

### 版本命名规范

```
Prompt-v{版本号}-{日期}-{作者}.md

示例：
Prompt-v1.0-20260307-ZhangSan.md
Prompt-v2.1-20260307-ZhangSan.md
```

- 初始版本

## 📈 效果评估框架

### 定性评估

| 维度 | 优秀 | 良好 | 待改进 |
|------|------|------|--------|
| **准确性** | 完全正确 | 基本正确 | 存在错误 |
| **完整性** | 覆盖全面 | 主要覆盖 | 遗漏关键点 |
| **一致性** | 多次稳定 | 基本稳定 | 波动较大 |
| **可用性** | 可直接使用 | 少量修改 | 大量修改 |

### 定量指标

- **准确率**：正确输出次数 / 总测试次数
- **一致性**：输出相似度（使用向量相似度）
- **Token 效率**：有效信息量 / Token 数量
- **时间节省**：人工耗时减少比例
