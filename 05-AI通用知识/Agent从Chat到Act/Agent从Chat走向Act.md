# Agent从Chat走向Act

## 为什么需要Act类Agent？

### 从Chat到Act的跨越
- Chat类Agent：帮我们梳理资料、回答问题
- Act类Agent：操作软件系统，如订外卖、改送货时间

### 业界动向
- **阿里CEO吴永铭**：深层次AI最大的想象力绝对是接管数字世界，改变物理世界
- **OpenAI演示**：Agent帮主人订咖啡
- **蚂蚁支小宝**：送奶茶到现场

### Act的本质
过去几十年互联网构建了巨大的数字世界，而AI带给我们的，是把人类从数字世界中解放出来，让Agent代替人类完成链接。

## Agent Act能力的两种实现方式

### 方式一：模拟人类操作数字世界

**原理：** 人类怎么操作，Agent就怎么操作

**案例：订机票步骤**
1. 提取出发地、目的地、出发日期
2. 找到网站字段并填入信息
3. 点击确认，提取航班信息
4. 返回给用户选择
5. 收到选择后点击确认
6. 填入乘机人信息
7. 点击提交订单
8. 请求账户密码
9. 点击确认完成

**问题：**
- 准确率低：Claude在开源数据集测试仅14.9%
- Token消耗大：一条指令约40万Token，成本3.2美元
- 微调后准确率也不超80%

### 方式二：调用数字世界接口（Function calling）

**原理：** Agent直接操作接口或函数

**案例：订机票步骤**
1. 调用查询机票接口
2. 返回航班信息给用户选择
3. 调用订单提交接口
4. 调用支付接口

**优势：**
- 准确率高：OpenAI函数调用准确率约85%
- Token消耗低：约1.2万Token

### Token消耗对比

```
Function calling Token消耗 = (API接口定义描述 + 自然语言指令) × (错误次数 + 1)
```

### 选择建议
- 优先选择有完善接口的"数字世界"
- 函数调用方式更具潜力

## 自助工单小助手案例

### 产品目标
让Agent在帮助车主解答问题的同时，根据故障情况安排师傅上门维修。

### 技术架构
- 意图识别 → TicketCreateAPI
- RAG知识库查询
- Re-Act模式：Think → Action → Observation

### Agent需要的能力
1. **准确的意图识别、Query重写和路由能力**
2. **让Agent正确地思考**：workflow学习
3. **让Agent正确地行动**：function calling学习

### 与Chat类Agent的区别
- Chat类Agent：侧重提示词工程、RAG
- Act类Agent：额外需要function calling能力
