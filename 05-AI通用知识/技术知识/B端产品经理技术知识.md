# B端产品经理技术知识

## 9.2 产品经理是否要懂技术

### 懂技术的价值

| 价值 | 说明 |
|------|------|
| 避免产品过度设计 | 避免价值不大的功能耗费大量开发资源 |
| 避免技术过度设计 | 砍掉不必要的代码灵活性和复杂性设计 |
| 顺畅沟通 | 与技术人员在同一频道上用同一套语言交流 |
| 预判可行性 | 快速判断技术实现的可行性和成本 |
| 评估工时 | 判断技术人员工时评估是否合理 |

### 常见问题：产品过度设计

**案例：热词配置功能**
- 产品经理设计复杂的后台配置界面
- 需要设计数据库、增删改查、排序功能
- 研发评估20人日

**简化方案**
- 只需一个文本框，一行一个词
- 不用设计数据库，保存为文本文件
- 只需3.5人日

**核心教训**
- 产品经理对技术一无所知导致过度设计
- 简化方案完全满足业务需求

## 9.4 B端产品经理的技术知识要求

### 9.4.1 基本技术知识体系

#### 理解编程语言
- 理解程序设计的基本逻辑
- 重点不是编写程序，而是理解原理
- 推荐从Java或C++入门

#### 掌握SQL
- **SQL**: Structured Query Language
- 关系数据库处理语言
- 实用价值：数据分析、数据处理

#### 网络通信常识
- TCP/IP协议 vs UDP协议
- 二进制、十六进制运算
- 对称密钥 vs 非对称密钥密码体系
- HTTPS vs HTTP安全性

### 9.4.2 MVC程序设计模式

#### MVC定义
- **M**odeling: 数据模型
- **V**iew: 前端交互视图
- **C**ontroller: 业务逻辑

#### 三层结构

| 层级 | 说明 | 技术示例 |
|------|------|----------|
| 前端交互层 | 绘制程序界面，完成用户交互 | JavaScript, HTML5, Vue |
| 业务逻辑层 | 处理业务逻辑 | Java, C++, PHP |
| 数据层 | 底层数据存储 | 关系数据库 |

#### 前端交互层
- 负责界面绘制和用户交互
- 实现简单的业务逻辑如数据校验
- 技术迭代快

#### 业务逻辑层
- 处理复杂业务规则
- 封装校验、判断、业务规则
- 与前端解耦，支持多端复用

#### 数据层
- 结构化/非结构化数据存储
- 通过程序对数据库进行增删改查

### 前后端分离架构

| 架构 | 说明 |
|------|------|
| 客户端 | 前端工程师负责 |
| 服务端 | 后端工程师负责（业务逻辑层+数据层） |
| 接口交互 | 客户端和服务端通过API接口通信 |

## 9.3 是否要关注技术方案

### 需关注的情况

| 情况 | 说明 |
|------|------|
| 技术方案影响产品方案 | 技术选型直接影响产品设计 |
| 技术方案导致项目风险 | 采用小众技术框架可能带来风险 |

### 不需干预的情况
- 技术人员专业范围内的技术方案设计
- 产品经理不具备足够的技术判断能力

### 核心原则
- 专业分工：决定技术方案是技术人员的分内之事
- 警惕：对技术人员的工作指指点点会让对方反感
