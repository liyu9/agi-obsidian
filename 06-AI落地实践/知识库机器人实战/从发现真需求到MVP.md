# 从发现真需求到MVP

## 发现需求：在了解"锤子"基础上找"钉子"

### 案例：家电消费品使用问答助手

**用户痛点发现过程：**
1. 热水器故障 → 找不到说明书 → 求助客服，花了十几分钟
2. 音响设备问题 → 说明书被扔掉 → 求助微信公众号客服
3. 说明书被抛弃是常态

**企业级痛点验证：**
- 年出货量40万台的电子消费品公司
- 客服特点：
  - 80%用户问题是产品如何使用的基本问题
  - 市场AI方案价格昂贵且效果不好
  - 需要雇佣10名全职客服，流动性高
  - 长尾诉求分析耗时太长

**向上抽象：**
只要有产品说明书的地方都存在这个诉求（to C硬件、轿车、to B生产设备）

## 解决方案设计

### RAG适合的场景

用户排除故障的过程：找到产品说明书 → 查到对应目录 → 理解内容 → 尝试方案

本质就是RAG的过程："检索 → 理解 → 输出"

### 三个端侧 + 两个功能模块

**三个端侧：**
| 端侧 | 用户 |
|------|------|
| 用户端 | 家电消费者 |
| 企业售后端 | 客服人员、管理人员 |
| LLM AI应用端 | 调用大模型能力 |

**两个功能模块：**
1. 回答产品说明书相关问题
2. 处理长尾诉求（转人工+AI辅助审核）

### 长尾诉求处理

- 人工客服回答后，用大模型为回答打分
- 用大模型给用户问题打标签，辅助分析统计

## 实现方法：Dify搭建MVP

Dify是大语言模型应用开发框架，类似电脑机箱：

**核心模块：**
| 模块 | 作用 |
|------|------|
| 知识库 | 上传产品说明书，设置检索规则 |
| Workflow | 拖拽连接各节点，控制流程 |

### AI时代的变化

- 产品经理可以不用代码，借助Coze、Dify等平台直接搭出MVP
- LLM AI改变了产品开发范式

### 三句总结

1. AI产品经理必须对技术有深度理解，才能发现需求
2. 从没能被上一代AI满足的需求中，尝试用新一代AI解决
3. AI产品经理不用代码也能搭建产品雏形，大大加速研发进程
