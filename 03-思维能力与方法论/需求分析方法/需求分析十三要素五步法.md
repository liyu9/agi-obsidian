# 需求分析十三要素五步法

## 6.2 通过十三要素五步法进行需求分析

### 背景案例：CRM消息中心优化失败

**失败教训**
- 产品经理果冻没有深入分析需求
- 只充当了"原型工人"
- 没有挖掘需求的真实痛点
- 最终做出的功能没人使用

**问题分析**
- 不熟悉业务，无法和需求方平等对话
- 豌豆是需求传递者，不是真正源头
- 没有认真分析需求、理解需求
- 混淆了个人待办管理和消息中心的概念

## 十三要素五步法

| 步骤 | 名称 | 覆盖要素 |
|------|------|----------|
| 步骤一 | 分析相关角色 | 提出人、使用人、受影响人 |
| 步骤二 | 了解基本场景 | 基本场景、发生频率 |
| 步骤三 | 挖掘真实动机 | 真实动机、解决动机 |
| 步骤四 | 发散更多场景 | 更多场景 |
| 步骤五 | 设计产品方案 | 产品方案 |

## 步骤一：分析相关角色

### 要素1：提出人
- 原始需求的提出人，而非传递人员
- 在公司中任何人都可能是提出人
- 提出人不一定是使用人
- 痛点不一定是提出人的痛点

### 要素2：使用人
- 最终使用产品功能的人
- 产品功能的终端用户
- 不一定是需求提出人

### 要素3：受影响人
- 最终受到产品功能影响的人
- 不一定是产品使用人

**案例分析**
- CRM需求中：提出人是销售（豌豆是传递者）
- 使用人是一线销售人员
- 受影响人也是一线销售人员

**扩展案例：线索分配需求**
- 提出人是销售部门老板
- 使用人是销售运营（配置功能给运营用）
- 受影响人是一线销售和主管

## 步骤二：了解基本场景

### 要素4：基本场景
**场景还原公式**
```
基本场景 = 人物 + 时间 + 地点 + 起因 + 经过 + 结果
```

**CRM案例场景还原**
> 电话销售同学栗子（人物），来公司半年了，每天勤勤恳恳，手里有不少老线索，每天也会分到很多新线索。在每个工作日的早上、下午和晚上（时间），坐在格子间的工位（地点），开着CRM系统的同时，要忙碌地联系不同客户。然而，今天下午，栗子同学又发脾气了，又有一条优质线索，因为不小心忘了跟进而掉库了（起因）！"这个系统一点儿都不好用！"（经过）总之能方便自己准确管理和及时跟进，不要漏掉每一条即将掉库的线索！（结果）

### 要素5：发生频率
- 了解需求背后事情的发生频率
- 高频需求优先级高
- 低频需求可考虑线下支持或低优先级

**例外情况**
- 即使频率低但重要的功能也要实现
- 例如：财务月结工作，每年只一次但很重要

## 步骤三：挖掘真实动机

### 黄金思维圈理论 (Simon Sinek)
- **Why**：为什么做
- **How**：如何做
- **What**：做什么

**核心观点**
- 从Why层出发，更容易洞悉问题本质
- 不要只停留在What层（表面需求）

### 要素6：真实动机
- 用户真正想解决的问题
- 需求背后的本质诉求

### 要素7：解决动机
- 提出人期望的解决方案
- 用户自以为正确的解决方案

## 步骤四：发散更多场景

### 要素8：更多场景
- 不要只聚焦于当前需求
- 发散思维，想到更多相关场景
- 全面覆盖用户需求

## 步骤五：设计产品方案

### 要素9-13：产品方案
- 基于前四步分析
- 设计全面的产品解决方案
- 覆盖所有相关场景

## 需求分析的正确态度

| 态度 | 说明 |
|------|------|
| 帮助用户 | 不是苛求用户 |
| 引导用户 | 挖掘真实诉求 |
| 专业能力 | 找到真正的问题点 |
| 不要传声筒 | 用户提什么就做什么 |

**核心观点**
- 产品经理有责任帮助用户找到真正的问题
- 而不是充当"原型工人"
- 要通过专业能力完成需求采集工作
