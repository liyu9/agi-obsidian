# 以用户为中心的体验设计

## 7.2 交互设计中的尼尔森十大原则

### Nielsen十大原则概述

| 原则 | 英文名 | 核心观点 |
|------|--------|----------|
| 系统可见性 | Visibility of system status | 系统应让用户始终了解发生了什么 |
| 系统与现实匹配 | Match between system and the real world | 使用用户熟悉的语言和概念 |
| 用户控制与自由 | User control and freedom | 提供明确的"出口"和撤销功能 |
| 一致性与标准化 | Consistency and standards | 保持界面和操作的统一性 |
| 防错原则 | Error prevention | 预防错误，而非出错后提示 |
| 记忆原则 | Recognition rather than recall | 减少用户记忆负担 |
| 灵活易用 | Flexibility and efficiency of use | 兼顾新手和高级用户 |
| 简约设计 | Aesthetic and minimalist design | 去除无关信息，突出重点 |
| 容错原则 | Help users recognize,diagnose,and recover from errors | 错误信息要通俗易懂 |
| 帮助原则 | Help and documentation | 提供必要的帮助文档 |

## 7.2.1 系统可见性原则

**核心观点**
- 系统应让用户始终了解发生了什么
- 及时反馈用户操作结果

**应用示例**
- 进度条显示上传/下载进度
- 按钮点击后显示加载状态
- 订单提交后显示处理状态

## 7.2.2 系统与现实匹配原则

**核心观点**
- 使用用户熟悉的语言和概念
- 遵循现实世界的习惯

**应用示例**
- 购物车图标表示采购
- 文件夹图标表示分类存储
- 日历图标表示时间选择

## 7.2.3 用户控制与自由原则

**核心观点**
- 用户常会误选功能，需要明确"出口"
- 提供撤销和重做功能

**应用示例**
- 编辑软件的撤销功能
- 删除确认对话框
- 电商平台允许取消订单

## 7.2.4 一致性与标准化原则

**核心观点**
- 同样的操作应有同样的结果
- 保持系统风格统一

**应用示例**
- App底部导航：首页排第一，个人中心排最后
- Office套装软件保持统一的菜单布局
- 相同功能使用相同的图标

## 7.2.5 防错原则

**核心观点**
- 预防错误发生，好过出错后再提示
- 设计时首先考虑如何避免错误

**应用示例**
- 按钮点击后置灰防止重复提交
- 调整按钮顺序防止误点
- 删除操作前二次确认

## 7.2.6 记忆原则

**核心观点**
- 计算机应减轻用户记忆负担
- 在需要时呈现相关信息

**应用示例**
- 搜索历史记录
- 地址簿保存常用地址
- 表单自动填充

## 7.2.7 灵活易用原则

**核心观点**
- 为大多数人设计，兼顾少数人
- 提供个性化定制

**应用示例**
- 软件快捷键设置
- 页面布局自定义
- Word自定义功能

## 7.2.8 简约设计原则

**核心观点**
- 对话中不包含无关信息
- 突出重点，避免信息过载

**应用示例**
- 停机通知突出停机时间和范围
- 表单只显示必要字段
- 报表突出关键数据

## 7.2.9 容错原则

**核心观点**
- 错误信息要通俗易懂
- 提供解决建议

**应用示例**
- 404页面解释"页面不存在"并给出建议
- 表单错误即时提示并说明规范
- 错误信息给出解决步骤

## 7.2.10 帮助原则

**核心观点**
- 系统应易于使用，不需要文档
- 但仍应提供帮助文档

**B端产品特点**
- B端产品蕴含复杂业务流程规则
- 用户自助服务难度高
- 帮助文档有存在必要

## 7.3 用户故事地图

### 核心概念
- 从用户视角分析使用产品的关键步骤和动线
- 基于场景拆解功能设计

### 用户故事地图结构

| 层级 | 说明 | 示例 |
|------|------|------|
| Activity | 一级场景 | 下单采购 |
| Backbone | 二级场景（骨架） | 选择商品→确认订单→支付 |
| User Task | 用户任务 | 选择商品、填写数量 |

### 用户故事格式
```
作为（谁），我想要（什么），以便（为什么）
As who, I want what, so that why
```

### 用途
- 敏捷开发中的功能拆分
- 迭代计划的制定
- MVP版本规划

## 7.4 客户旅程地图

### 核心概念
- 分析用户在完整使用产品过程中的动作
- 对动作节点进行情绪分析

### 客户旅程地图结构

| 区域 | 要素 | 说明 |
|------|------|------|
| A | 客户画像 | 谁 |
| A | 核心场景 | 要做什么 |
| B | 子场景 | 细分场景 |
| B | 分解动作 | 具体操作 |
| B | 客户想法 | 用户思考 |
| B | 情绪变化 | 满意度 |
| C | 优化空间 | 改进点 |
| C | 责任人 | 跟进人员 |

## 7.5 服务蓝图

### 核心概念
- 研究服务提供者和客户之间的接触点、流程
- 关注旅程节点背后的企业内部运作

### 服务蓝图结构

| 层级 | 说明 |
|------|------|
| 客户旅程 | 核心骨架，描述客户达成目标所经历的旅程 |
| 发生现场 | 客户使用服务的场所（数字产品或物理场所） |
| 前台 | 企业与客户直接触点（员工动作、技术触点） |
| 后台 | 支持前台业务运作的内部运作过程 |
| 后台系统 | 软件系统或复杂业务流程 |

### 服务蓝图要素
- 耗时(Time)：各节点花费时间
- 交互分界线：客户行为与企业行为的分界
- 感知分界线：客户有感知与无感知事件的分界
- 内部动作分界线：前后台运作模块的分界

### B端应用场景
- 适合分析客户服务体验
- 帮助产品经理以客户为中心分析问题
- 呈现企业运作全貌
