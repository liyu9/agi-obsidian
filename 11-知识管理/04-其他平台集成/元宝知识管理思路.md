### **可落地的四阶段解决方案：用 Obsidian + AI 构建产品设计知识管理中枢**

**核心目标**：将"写文档"从一项零散、高成本的维护工作，转变为"知识沉淀"的自然结果，实现文档的**结构化、原子化、版本化、自动化**。

#### **阶段一：奠基与准备（解决"无标准、归集难"）**

此阶段目标是建立统一的知识库结构和输入规范，为AI和后续工作打好基础。

1. **在Obsidian中建立核心知识库（Vault）**：

    - **单一数据源**：规定所有最终版、需留痕的产品设计知识，都必须存储于此。杜绝分散存储。

    - **目录结构设计**：

        ```
        产品知识库/
        ├── 0-产品总览/           # 产品愿景、核心指标、演进路线图
        ├── 1-业务概念/           # 领域模型、核心实体、业务术语表
        ├── 2-需求池/             # 按模块或业务线组织的未来需求
        ├── 3-产品设计/
        │   ├── 【产品模块A】/
        │   │   ├── 01-需求背景与目标.md
        │   │   ├── 02-用户角色与场景.md
        │   │   ├── 03-功能列表与迭代地图.md
        │   │   └── 04-核心业务规则库/  # **关键**：原子化规则存放处
        │   ├── 【产品模块B】/
        │   └── 全局规则/           # 如权限体系、数据规范
        ├── 4-迭代版本/
        │   ├── v1.0-迭代名称/
        │   │   ├── 迭代需求清单.md
        │   │   └── 设计文档索引.md  # 不复制内容，只链接到"3-产品设计"中的具体模块和规则
        │   └── v2.0-迭代名称/
        ├── 5-会议与决策记录/      # 每次重要讨论、决策背景
        └── 模板/                  # 存放各类标准化模板
        ```

2. **定义"文档类型"与标准化模板**：

    - **在Obsidian中创建模板文件**：为`业务规则`、`需求背景`、`用户角色`、`迭代简报`等创建标准模板。模板中包含固定结构和示例。

    - **赋予AI识别能力**：将模板结构和您定义的"文档名、类型、说明"列表提供给trae，训练AI在辅助你记录时，能自动建议或套用对应模板格式。

3. **四大文档生命周期与12类核心文档（AI精准识别基础）**：

    - 按产品设计全流程，划定**4大文档生命周期**、**12类核心文档**，明确每类文档的定义、拆分粒度、必填字段，确保AI可精准分类识别：

    | 生命周期阶段 | 核心文档类型 | 文档说明 | 最小拆分粒度 |
    |------|-----------|---------|-----------|
    | 需求调研阶段 | 需求调研文档 | 用户访谈、市场分析、业务痛点、需求池清单 | 单需求/单用户角色为1个独立单元 |
    | 需求调研阶段 | 需求评审记录 | 需求评审结论、待办项、决策意见 | 单次评审为1个独立单元 |
    | 产品设计阶段 | PRD原子模块文档 | 业务背景、用户角色、核心业务规则、业务流程、异常场景、权限设计等 | 最小可独立变更单元（1条核心规则/1个独立流程/1组异常场景为1个独立文档） |
    | 产品设计阶段 | 流程/原型关联文档 | 业务流程图、页面流程图、高保真原型、UI规范 | 单流程/单模块原型为1个独立单元 |
    | 迭代落地阶段 | 迭代版本计划 | 迭代排期、需求范围、责任人、上线节点 | 单个迭代版本为1个独立单元 |
    | 迭代落地阶段 | 需求变更记录 | 变更背景、变更内容、影响范围、决策人、生效时间 | 单次变更为1个独立单元 |
    | 迭代落地阶段 | 测试用例关联文档 | 核心测试点、边界条件、回归用例 | 单功能模块为1个独立单元 |
    | 复盘沉淀阶段 | 上线复盘文档 | 上线效果、问题复盘、优化方向、数据结论 | 单次迭代/单个功能上线为1个独立单元 |
    | 复盘沉淀阶段 | 产品全局手册 | 业务全景、核心架构、全量规则、权限体系 | 按业务线/功能模块拆分独立单元 |
    | 复盘沉淀阶段 | 团队知识沉淀 | 踩坑记录、设计规范、复用组件、行业案例 | 单知识点为1个独立单元 |

4. **标准化YAML元数据规范（版本管控与检索核心）**：

    - 所有文档必须在Obsidian笔记开头固定配置YAML元数据，无例外。该规范是解决版本混乱、检索困难、AI精准匹配的核心：

    ```yaml
    ---
    文档唯一ID: 业务线缩写-文档类型-年月日-序号（例：trade-rule-20260227-001）
    文档类型: 对应上表的12类固定文档类型
    所属业务线: 归属业务/产品模块
    所属迭代版本: 语义化版本号（例：V2.3.0）
    生效状态: 草稿/评审中/已上线/已作废（4种固定状态，禁止自定义）
    创建人:
    创建时间: YYYY-MM-DD
    最新更新人:
    最新更新时间: YYYY-MM-DD
    关联文档ID: 所有关联的文档唯一ID，用逗号分隔
    关联需求ID: 对应需求池/项目管理平台的需求编号
    ---
    ```

#### **阶段二：原子化与结构化（解决"单文档庞杂、维护难"）**

这是破解PRD"牵一发动全身"困局的关键，将传统长文档拆解为可复用的知识单元。

1. **实施"原子化"写作原则**：

    - **核心规则独立成文**：在`04-核心业务规则库/`中，**每一条**重要的业务规则、异常场景、数据指标，都创建一个单独的Markdown文件（如`用户注册-手机号格式校验规则.md`）。

    - **使用唯一标识**：为每条规则设置ID（如`RULE-AUTH-001`），便于追踪。

    - **内容标准化**：每条规则文件包含：规则ID、描述、触发条件、处理逻辑、关联UI/接口、修订历史。

2. **用"链接"替代"复制粘贴"**：

    - 当需要描述一个完整功能时，不再大段文字叙述，而是创建一个"功能设计"文档，通过内部链接`[[ ]]`的方式，**嵌入或引用**相关的原子规则、用户角色、流程图。

    - **例如**：`"优惠券发放流程"文档中，关于"发放条件"部分，只需写："具体条件遵循 [[RULE-COUPON-101 发放资格规则]] 与 [[RULE-COUPON-102 发放时间规则]]"`。

3. **带来的好处**：

    - **一处改，处处新**：修改`RULE-COUPON-101`文件，所有链接引用了它的功能文档自动同步最新内容。

    - **结构清晰**：文档变成一张由知识点组成的网络，而非线性文本。

    - **杜绝细节丢失**：每个细节都是一个独立的、可被链接和搜索的节点。

#### **阶段三：AI辅助的日常工作流（解决"记录成本高、版本管理失控"）**

此阶段将trae等AI工具深度融入日常工作，让知识更新变得轻松。

1. **日常构思与记录**：

    - 在trae中自由讨论、脑暴，形成初步方案。**确定方案后，立即操作**：将讨论的结论，用AI按照Obsidian对应模板，整理成结构化内容。

    - **AI执行示例**：你可以对trae说："将我们刚刚确定的'任务审核失败'的三种情况，按照'业务规则模板'，整理成三条原子规则，并分配规则ID。然后，更新'任务管理模块'的设计文档，在'审核流程'部分链接这三条新规则。"

2. **版本与迭代管理**：

    - **迭代版本即目录**：每个新迭代，在`4-迭代版本/`下创建新文件夹（如`v2.3-搜索功能优化`）。

    - **迭代文档即索引**：该文件夹下的`设计文档索引.md`，不重复撰写内容，而是列出本次迭代**修改、新增、废弃**的原子规则和功能文档的链接列表，并说明变更原因。

    - **自动生成更新日志**：利用AI工具，对比本次迭代索引与上次迭代索引的链接差异，自动生成面向研发、测试的"需求变更点清单"。

3. **自动化知识入库**：

    - 定义好规则后，可以让AI将日常会议纪要、邮件决策中关于产品设计的结论，自动提取、格式化，并建议存入Obsidian知识库的对应位置，你只需审核确认。极大降低手动录入成本。

#### **阶段四：知识消费、协作与传承（解决"查找难、对齐难、新人上手难"）**

让沉淀的知识活起来，服务于协作和团队。

1. **利用Obsidian的图谱和搜索**：

    - **可视化关联**：通过"本地图谱"功能，新人可以直观看到核心功能由哪些规则构成，规则又被哪些功能引用，快速建立系统认知。

    - **全局搜索**：利用全文搜索和标签，秒级定位任何历史规则、决策背景。

2. **生成"动态产品手册"与协作视图**：

    - **一键发布**：使用Obsidian的发布插件，将`3-产品设计/`下的内容发布成一个内部网站，作为**唯一可信源**。研发、测试、设计直接访问此网站，永远是最新内容。

    - **为协作方生成视图**：利用AI，从知识库中提取某个迭代或功能的所有相关信息，为测试生成"测试点清单"，为研发生成"接口逻辑摘要"，实现精准信息分发。

3. **新人入职引导**：

    - 新人学习路径：阅读`0-产品总览`-> 浏览`1-业务概念`-> 通过图谱探索核心模块 -> 查看最新两个迭代的"设计文档索引"了解近期动态。知识是结构化的，学习是探索式的，效率远超阅读混乱的PRD。

### **总结：闭环工作流**

1. **思考与记录在AI工具**（如trae）：自然语言交互，自由构思。

2. **结构化与沉淀在Obsidian**：将结论原子化、模板化存入，形成知识网络。

3. **版本与协作通过目录和发布**：用链接和索引管理迭代，用发布功能同步团队。

4. **消费与检索通过图谱和搜索**：可视化理解，精准查找，高效传承。

这套方案直接命中您总结的所有痛点，将文档从"管理的负担"转化为"核心竞争力的知识资产"。
