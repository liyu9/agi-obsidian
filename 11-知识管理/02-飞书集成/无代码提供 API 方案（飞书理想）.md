# 分阶段 API 开发优化规划表

## 规划总原则

1. **能力贴合原则**：一期 100% 基于现有已上线产品能力（类多维表格应用、数据表 / 字段 / 记录管理、权限配置），不新增产品功能，仅做 API 标准化与权限绑定；二期 / 三期严格匹配产品视图、流程的迭代计划，同步开发对应 API。
2. **场景优先原则**：优先覆盖 1000 人规模高频 CRUD 场景，同时优先满足 dify、openclaw 集成的核心依赖 API，避免无效开发。
3. **权限强绑定原则**：100% 对标飞书「权限点 - API」强绑定逻辑，每个 API 唯一对应权限标识，调用前强制权限校验，适配企业级分级管控。
4. **兼容扩展原则**：API 命名、入参出参、分页规则完全对齐飞书 API 规范，预留扩展字段，最大化降低 dify/openclaw 的对接适配成本。

---

## 一期：基础闭环期（周期 1-2 个月）

### 本期核心目标

1. 完成现有已上线产品能力的 API 全量标准化，实现「应用 - 数据表 - 字段 - 记录 - 权限」全链路 API 闭环；
2. 建立企业级「权限点 - API」强绑定体系，适配 1000 人规模的分级权限管控；
3. 满足 dify、openclaw 集成的**最低核心依赖**，支撑基础 AI 数据读写、自动化执行场景；
4. 搭建标准化 API 网关、双鉴权体系、OpenAPI 文档，完成基础性能与安全保障。

### 本期配套保障

1. 鉴权体系：搭建「应用级 tenant_access_token + 用户身份 user_access_token」双鉴权体系，每个 API 调用强制校验对应权限点，无权限直接拦截；
2. 标准化规范：统一 API 命名、入参出参结构、分页规则、错误码体系，100% 对齐飞书 API 规范，输出 OpenAPI 3.0 标准可视化文档；
3. 性能管控：设置批量 API 阈值、单 IP / 应用限流规则，适配 1000 人规模并发，避免数据库压力；
4. 日志审计：全量 API 调用日志留存，支持权限越权、异常调用审计，满足企业合规要求。

## 配套对接 dify、openclaw 的分阶段策略

与 API 开发周期完全对齐，避免超前对接导致的返工：

1. 一期 API 上线后：完成 dify 基础集成（AI 数据读写、结构化数据生成）、openclaw 基础动作节点注册（手动触发型自动化）；
2. 二期 API 上线后：完成 dify 高阶场景集成（AI 视图 / 表单配置、附件内容解析）、openclaw 事件触发型自动化落地；
3. 三期 API 上线后：完成 dify+openclaw 深度整合，实现「自然语言指令→AI 生成配置→自动化流程执行」全链路闭环。
